13. **Tailwind動的クラス問題の解決（2026/01/07）**

    **問題:**
    Tailwind CSS v4では動的クラス名（`bg-blue-50/50`等）がデフォルトで含まれていない。

    **解決策:**
    `globals.css`にカスタムCSSクラスを定義：
    ```css
    /* 担当者別カード背景色 */
    .card-bg-blue { background-color: #bfdbfe; }
    .card-bg-pink { background-color: #fbcfe8; }
    .card-bg-yellow { background-color: #fef08a; }
    .card-bg-green { background-color: #bbf7d0; }
    .card-bg-cyan { background-color: #a5f3fc; }
    .card-bg-purple { background-color: #e9d5ff; }
    .card-bg-orange { background-color: #fed7aa; }
    .card-bg-gray { background-color: #d1d5db; }

    /* max-widthクラス（Tailwind v4で未定義） */
    .max-w-7xl { max-width: 80rem; }
    .max-w-8xl { max-width: 88rem; }
    ```

    **変更ファイル:**
    - `src/app/globals.css` - カスタムクラス追加
    - `src/lib/data/index.ts` - `cardBgColor` → `cardBgClass`に変更
    - `src/components/TaskCard.tsx` - インラインスタイル → クラス名使用

    **複数担当者の優先順位:**
    ```
    河合(1) > 川崎(2) > 中尾(3) > 下脇田(4) > 紺谷(5) > 清水(6) > 青柳(7) > 渡邉(99)
    ```

    - `getPriorityAssigneeColor()`関数を追加

16. **フローステップ表示の実装（2026/01/07）**

    **概要:**
    「備考・課題感」を「フロー」表示に置き換え。各業務を横並びのボックスで視覚化し、必要なフォーマット/シートへのリンクをステップ内に配置。

    **データ構造の追加（`src/lib/data/index.ts`）:**
    ```typescript
    interface FlowStepLink {
      label: string;           // "フォーマット"
      url?: string;            // リンクの場合のURL
      content?: string;        // ポップアップの場合のコンテンツ
      type: 'link' | 'popup';  // リンクか、ポップアップか
    }

    interface FlowStep {
      label: string;           // "LINEWORKSで企業グループを作成"
      links?: FlowStepLink[];  // このステップで使うリンク/ポップアップ
    }

    // Task に追加
    flowSteps?: FlowStep[];
    ```

    **UI変更（`src/components/TaskCard.tsx`）:**
    - 「備考・課題感」セクションを削除
    - 「フロー」セクションを追加（横並びボックス + 矢印）
    - ステップ内にリンク/ポップアップボタンを配置

    **表示イメージ:**

    ```
    フロー
    ┌──────────────────────┐    ┌──────────────────────┐    ┌──────────────────────┐
    │ ワークスで           │    │ フォーマットに記入   │    │ 担当者をメンション   │
    │ 企業グループを作成   │ →  │ [📄 フォーマット]    │ →  │ して送信             │
    ```

## 重要な用語定義

### 「トリガー」の概念（重要）

業務と業務の間には「トリガー」が必要。トリガー = 次の業務を開始するきっかけ・条件。

```
開始 → [トリガー] → 業務A → [トリガー] → 業務B → ... → 完了
```

例：
- 「ワークスで担当者メンション + 必要情報送信」→ 初回打ち合わせ日程調整開始
- 「議事録のワークスグループ共有」→ ヒアリング内容整理開始

### 「次の業務担当」の意味（変更あり）

**次の業務担当（nextAssignee）= 次の工程を担当する人**

- **担当**: 現在この業務を担当している人
- **次の業務担当**: この業務の次の工程を担当する人

※ 2026/01/07に「後任」から「次の業務の担当者」に意味を変更。

#### 次のタスク: 構成案生成GAS + プロンプト（2026/01/09設計完了）

**目的:**
ヒアリングシートの情報を入力すると、構成案テンプレートに沿った原稿案を自動生成する

**実装パターン:**
`transcriptToHearingSheet.js`と同じパターンで実装する
1. GAS: ヒアリングシートの情報を取得して出力
2. プロンプト: AIに構成案を生成させる
3. 結果: 構成案が完成

**フロー:**
```
ヒアリングシート（Part①+Part②）
    ↓ GAS「構成案を生成（プロンプト生成）」
ヒアリングシート情報 + プロンプト
    ↓ AIに投げる（Claude/Gemini）
構成案（原稿用 + 動画用）
    ↓
ワークスで共有
    ├→ 原稿執筆（河合, 中尾文香）  ← 原稿用の情報を使用
    └→ 企画・質問設計（川崎）      ← 動画用の情報を使用
         ↓
       撮影 → 編集
```

**ポイント: 並行作業への対応**
構成案完成後、原稿チームと動画チームが並行で作業を開始する。
そのため、構成案には**両方に必要な情報**が含まれている必要がある。

**ヒアリングシートの構造:**
```
Part① 基本情報（企業様ご記入 = 黄色セル）
├─ 企業概要：企業名、代表者、HP、住所、連絡先、事業内容、転勤
├─ 雇用形態：正社員/契約等、試用期間、職種
├─ 勤務時間：勤務時間①②③、実働時間
├─ 休憩時間
├─ 残業：有無、月平均
├─ 休日：休日区分、年間休日、長期休暇
├─ 給与：給与区分、給与額、想定年収、賞与、みなし残業
├─ 待遇・福利厚生：社会保険、その他待遇
├─ 作業内容：作業内容詳細
├─ 製品・商品：サービス・製品①②③、注意点
└─ その他：平均年齢・男女比、必須資格、選考フロー

Part② ヒアリング情報（打ち合わせ時記入 = 青色セル）
├─ 会社紹介：私たちについて、社長挨拶、会社の魅力、雰囲気
├─ 社員の声：①②③④（氏名、部署、年数、インタビュー内容）
├─ 求人写真：チェック（若手社員、女性社員、製品・部品、現場など）
├─ ★最も打ち出したいポイント★
├─ 募集情報：募集背景、ペルソナ（性別、年齢、外国人）、求める人材像
└─ スカウトメール設定
```

**構成案に含めるべき情報:**

| 用途 | 元データ | 構成案での項目 |
|------|---------|---------------|
| **原稿用** | Part①全般 | 求人概要（給与、勤務、休日等） |
| | Part②「私たちについて」 | 会社紹介 |
| | Part①「作業内容詳細」 | 業務内容Step形式 |
| | Part②「会社の魅力」「最も打ち出したいポイント」 | 会社のポイント×3 |
| | AIで生成 | キャッチコピー案 |
| | Part②「求人写真」 | 掲載画像の方向性 |
| **動画用** | Part②「最も打ち出したいポイント」 | 訴求ポイント |
| | Part②「社員の声」 | インタビュー対象者 |
| | AIで生成 | インタビュー質問案 |
| | Part②「求人写真」チェック | 撮影で押さえるシーン |
| **共通** | Part②「ペルソナ」 | ターゲット層 |
| | Part②「会社の魅力」等 | 差別化ポイント |

**作成するもの:**
1. **GAS: ヒアリングシート情報取得**
   - Part①+Part②の情報を取得
   - プロンプトと組み合わせてクリップボードにコピー
   - `hearingSheetManager.js`に統合 or 新規GASファイル

2. **プロンプト: 構成案生成用**
   - 入力: ヒアリングシートの情報（JSON形式）
   - 出力: 構成案（原稿用セクション + 動画用セクション）

**参考データ:**
- コベルク様の完成原稿（ペアソナ/エンゲージ）
- 構成案テンプレート（tsunageru.ts内format）

1. 社長がこの会社を好きな理由

4. 社長が「ここは誇っていい」と思っているところ

6. この会社らしい働き方
7. 応募者に伝えたい「安心できるポイント」
8. それでも大事にしていること
9. この会社に向いている人
10. 一言メッセージ

**社員インタビュー（10問）:**
1. 働く理由の核
2. この環境を選んだ理由
3. 入ってみて分かった現実
4. 折れなかった理由（定着の正体）
5. 変化した自分
6. 活躍している人の共通点（行動レベルで3つ）
7. 合わない人の特徴（逆ペルソナ）
8. この環境の正体（比喩）
9. 応募者への正直な一言
10. それでも来てほしい人

**構成案との関係:**
- 構成案 = 主に原稿用
- 動画用の質問 = 固定テンプレート
- 構成案から必要なのは「誰にインタビューするか」（インタビュー対象者）

## 次世代セッションへの引き継ぎ事項（重要）

### 優先度高: hasInputField付きpopupの複数入力対応

**現状の問題:**
- ContentModalの`{{input}}`は1つの入力フィールドの値のみ対応
- 撮影連絡など、企業名・日付・場所等を別々に入力したいケースで不便

**対応方針:**
1. ContentModal.tsxを拡張して複数の入力フィールドをサポート
   - `{{input1}}`, `{{input2}}`, `{{input3}}`等のプレースホルダー対応
   - FlowStepLink型の拡張（inputFields配列等）
2. または、GASダイアログ側で対応（設定シートとの連携で担当者名を動的置換）

**関連ファイル:**
- `src/components/ContentModal.tsx`
- `src/lib/data/index.ts`（FlowStepLink型）
- `docs/gas/setupSheets.js`（設定シート・プレースホルダー置換）

### 優先度高: @担当者名のGAS設定シート連動

**現状の問題:**
- テンプレート内の`@河合 @中尾文香 cc:@青柳`等がハードコード
- 担当者変更時にコード修正が必要

**既存の実装（setupSheets.js）:**
- 設定シートに担当者情報を保存
- `{{原稿担当1}}`, `{{@原稿チーム}}`, `{{CC}}`等のプレースホルダーを用意
- `replacePlaceholders()`関数で置換

**対応方針:**
1. GASダイアログのプロンプトシート機能を活用
2. Next.js側のhasInputField付きpopupはハードコードのまま使用し、コピー後に手動修正を許容
3. または、Next.jsからGAS APIを呼び出して設定シートの値を取得（複雑）
